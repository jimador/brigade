# AGENTS.md — the brigade plugin repo (brigade cooking itself)

Repo-level facts for agents working on Brigade.

## Brigade configuration

- **Source:** `local` (adapter: `skills/brigade/sources/local.md`) — board under
  `.brigade/board/`, git-excluded with the rest of `.brigade/`.
- **Operator:** the human running the session (`assignee` frontmatter).
- **Status mapping:** identity (`backlog`/`scoping`/`design`/`todo`/`in_progress`/
  `in_review`/`done`/`blocked`).
- **Verification gate:** `./test/regression.sh` plus the syntax and bundle checks listed
  in `CONTRIBUTING.md` (and anything in `.brigade/config.md` when present).
- **Dishes:** `.brigade/dishes/<slug>/` is the control plane; the board mirrors status.

## Custom fields (markdown frontmatter)

- `kind` — `feature | bug | chore | docs | research | contract`
- `worker` — cook roster name at dispatch; empty before claim

## What this repo ships

Markdown is the product. Skills and agent files are prompts that run in every session that
installs the plugin — treat their wording with the care you would give production code.

`workflows/brigade-*.js` are generated by `bin/brigade-bundle` from `workflows/src/*.js` +
`workflows/config.js`. Never hand-edit the generated files.

Tier policy lives in both `skills/brigade/TIERS.md` and `workflows/config.js`; artifact
schemas in both `skills/brigade/SCHEMAS.md` and `bin/brigade-validate`. Each pair changes
together, in the same commit.

## Guardrails

- Never commit `.brigade/` (it is in `.git/info/exclude`).
- Design swag never claims tickets or sets `worker`.
- No org-specific adapters, no secrets, and no real names, handles, repos, orgs, emails,
  or absolute home paths — anywhere, including examples and fixtures.
- Scripts stay dependency-free and BSD/macOS compatible; no `jq` requirement in a code
  path that must work without it.
- Never use "mise" / "mise en place" vocabulary — the `mise` dev-tool manager owns it.
